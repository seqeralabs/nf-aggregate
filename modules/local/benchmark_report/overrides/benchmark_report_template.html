Methodology — what each utilization metric means

The report exposes four different utilization metrics. They sound similar but measure very different things, and in practice they disagree by 2–5×. Read this before comparing rows in the run-metrics table below.

Metric Level Resource Numerator Denominator What it captures
Trace efficiency (task)
cpuEfficiency / memoryEfficiency		 Task CPU Σtasks (pcpu / 100 · realtime) Σtasks (cpusreq · realtime) How close each task came to using what it asked for. Ignores VM packing and VM idle time.
Memory Σtasks (peakRss · realtime) Σtasks (memoryreq · realtime)
Sched allocation (VM)
schedAllocCpuEfficiency / schedAllocMemEfficiency		 VM CPU ΣVMs (avg_cpu_util · vCPU · lifetime) ΣVMs (vCPU · lifetime) How booked each VM was. Per-VM avg_util = Σ(requesttask · task_duration) / (capacityVM · vm_lifetime). Drops when a VM sits idle between tasks or when small tasks run on large instances.
Memory ΣVMs (avg_mem_util · memoryVM · lifetime) ΣVMs (memoryVM · lifetime)
Real VM efficiency (VM)
realVmCpuEfficiency / realVmMemEfficiency		 VM CPU Σtasks (pcpu / 100 · realtime) ΣVMs (vCPU · lifetime) Ground truth: what fraction of the VM-hours you paid for was actually spent doing work.
Memory Σtasks (peakRss · realtime) ΣVMs (memoryVM · lifetime)

How to read them: Trace efficiency tells you whether users are requesting too much CPU/memory per task. Sched allocation tells you how well the scheduler packs and reuses VMs. Real VM efficiency is the end-to-end ground truth: what fraction of the VM-hours you paid for were actually spent doing work. Trace efficiency × Sched allocation ≈ Real VM efficiency (modulo time-weighting).

VM-level metrics require a machines.jsonl in the JSONL bundle (built from a per-run machine CSV — sched UI /machines export or the instance_summary.csv produced by analyze_idle_periods.py). Runs without machine data show — for those columns.

Performance gains

Summary
Decomposes the VM capacity-hours you paid for into used vs wasted, and attributes the waste to the three responsibilities the Seqera scheduler owns: VM provisioning (choosing instance types), packing (which task on which VM and when), and task rightsizing (trimming user-requested CPU/Memory before submitting to ECS, when prediction is enabled).

Four quantities, per run
VM capacity: ΣVMs (vCPU · lifetime). What you paid for.
User requested: Σtasks (cpusconfig · realtime). Nextflow-directive request per task × active time.
Scheduler booked: schedAllocCpuEfficiency · VM capacity. What actually went to ECS as cpuShares (reflects Predv1 rightsizing when on).
Used: Σtasks (pcpu/100 · realtime). Actual work done.
Memory analogues use memoryconfig, schedAllocMemEfficiency, and peakRss.

How the scheduler's three jobs show up
   — Rightsizing shrinks scheduler booked below user requested. Visible as the green “already saved” bar in the savings chart.
   — Packing shrinks the gap between scheduler booked and VM capacity. Visible as “VM slack” in the capacity mix.
   — Provisioning shrinks VM capacity itself (picking smaller/cheaper instances). Visible as a shorter total bar.
Rightsizing does not directly raise Real VM efficiency — it only frees VM headroom for packing/provisioning to consume.

▶ Scheduler strategy by run group

Note on Predv1 visibility. The per-task cpus value in the Nextflow trace is always the Nextflow config request, not the scheduler-submitted value. Rightsizing savings are therefore derived from aggregate /machines utilization, which is lossy: when Predv1’s rightsizing is small relative to scheduler-side overhead (ramp-up, handoff between tasks on the same VM, concurrent-task edge cases), the aggregate can show scheduler booked ≥ user requested and the rightsize layer clamps to 0. This is an under-count, not a correctness bug — per-task scheduler-booked CPU / memory from a /v1a1/compute/tasks/{id} response would make it exact. A staging/retry layer using duration is intentionally omitted — on spot runs it silently bundles staging with interrupted-VM and retry-queue time.
